session与cookie得区别

Date 周一 20 十一月 2017 By 浮绿水 Category 技术. Tags pelican publishing

HTTP协议的特性：

每一次来自于用户浏览器的请求（request）都是无状态的、独立的。通俗地说，就是无法保存用户状态，后台服务器根本就不知道当前请求和以前及以后请求是否来自同一用户。

为了保持连接状态，网站会在用户的浏览器通过Cookie可以保存一些诸如用户名、浏览记录、登录和注销等各种数据。但是这种方式非常不安全，因为Cookie保存在用户的机器上，如果Cookie被伪造、篡改或删除，就会造成极大的安全威胁，因此，通常将Cookie用来保存一些不重要的内容，实际的用户数据和状态还是以Session会话的方式保存在服务器端。

cookie：在本地客户端存储数据，可以发送给服务端

Cookie：指某些网站为了辨别用户身份、进行会话跟踪而储存在用户本地的数据（通常经过加密）。 Cookie是由服务器端生成，发送给客户端浏览器，浏览器会将Cookie的key/value保存，下次请求同一网站时就发送该Cookie给服务器（前提是浏览器设置为启用cookie）。 Cookie可以长久的在客户端保存，如购物车

session ：会话~~在服务端存储数据，安全，存储数据多，但有有效期

Session：对于敏感重要的信息，建议要存储在服务器端，不能存储在浏览器中，如用户名、余额、等级、验证码等信息 Django的session默认保存14天，默认创建在django数据库 Session是大多数网站都需要具备的功能。Django为我们提供了一个通用的Session框架，并且可以使用多种session数据的保存方式： 保存在数据库内 保存到缓存 保存到文件内 保存到cookie内

cookie与session的区别： Session就是在服务器端的‘Cookie’，将用户数据保存在服务器端，远比保存在用户端要安全、方便和快捷得多。 Session依赖于Cookie！但与Cookie不同的地方在于Session将所有的数据都放在服务器端，用户浏览器的Cookie中只会保存一个非明文的识别信息，比如哈希值。

检查配置： Django通过一个内置中间件来实现会话功能。要启用会话就要先启用该中间件。 编辑settings.py中的MIDDLEWARE设置 确保存在django.contrib.sessions.middleware.SessionMiddleware这一行。 【默认情况在新建的项目中它是存在的。】 如果不想使用会话功能，那么在settings文件中， 将SessionMiddleware从MIDDLEWARE中删除， 将django.contrib.sessions从INSTALLED_APPS中删除就OK了。

  1. 基于数据库的会话 确保在INSTALLED_APPS设置中django.contrib.sessions的存在，然后运行manage.py migrate命令在数据库内创建sessions表。默认情况下，Django将会话数据保存在数据库内

cookie： set_cookie() 设置cookie（不识别中文,存储的为中文的编码） delete_cookie() 删除cookie request.COOKIES.get() 获取cookie request.set_signed_cookie() 添加签名设置cookie（用于防止用户在客户端进行修改cookie）
request.get_signed_cookie() 添加签名获取cookie 参数 salt = '' 给 cookie 值添加签名,得到一个通过salt盐值加了签名的 Cookie 值 其他参数与set_ccookie相同 使用set_signed_cookie时必须把值前面的参数写完整防止程序混乱 使用get_signed_cookie时如果对应的key值不存在则会引发 KeyError 异常 所以要记得异常捕获来确定是否含有 cookie 值

set_cookie中的几个参数： key = 'cookie的键' value= '对应此键的值' secure=False ：https 加密传输设置，当使用 https 协议时，需要设置该值为True。 path='/' ：指定哪些url可以访问到cookie，默认'/'为所有。 max_age=None ：cookie生效的时间，单位为秒，值为时None只持续在客户端浏览器的会话时长 存在该值时， expires 会被计算得到。 expires=None ：cookie过期的日期，是一个 datetime.datetime 对象， 该值存在时，那么max_age 也会被计算得到。 domain=None ：用于设置一个跨域的cookie值。 如domain='.test.com' ，可以被 www.test.com 、 bbs.test.com 等主域名相同的域读取为None时， 代表当前域名下全局生效。 httponly=False ：HTTPOnly 是包含在 HTTP 响应头部中 Set-Cookie 中的一个标记。 为一个bool值，当设置为True时，代表阻止客户端的 Javascript 访问 Cookie 。 这是一种降低客户端脚本访问受保护的Cookie数据风险的有效的办法。

session: request.session['键'] = '值' ：设置session request.session.get('键') ：从数据库中查找键并获取值如不存在该 Key 值，不会引发异常，返回 None request.session['键'] ：如 Key 值不存在，引发 KeyErro del request.session['键'] ：删除单条session值Key 值不存在时，引发 KeyError request.session.clear() ：删除所有session值 request.session.flush() ：清空session数据库

request.session.set_expiry() ：设置session的超时时间 request.session.get_expiry_age() ：返回距离过期还剩下的秒数 request.session.clear_expired() ：清除过期的 Session 会话 为整数时 ：将在value为秒单位之后过期 为0时 ：将在用户关闭浏览器之后过期。 为None时 ：使用全局过期的设置，默认为两个星期，14天。 为datetime时 ：在这个指定时间后过期。

登录： if request.method == 'post': ：判断获取类型 try: users = Users.objects.get(user=user,password=password) ：判断获取到的数据是否符合数据库中的数据 except: ：有异常 表示获取不到或者不相等 失败 else: ：如果try中的语句没有引发异常，则执行else中的语句 成功
